The shared ingress

One Kubernetes cluster can host one or many IFS Cloud installations. A shared ingress proxy inspects incoming requests and routes each to the correct namespace — the isolation boundary for an environment.

Namespaces and containers

Each environment — production, test, configuration — is a namespace holding its own set of containers: the OData API back-end, IAM, client services, integration services. They are independently versioned and independently scaled.

Why containers

Containerisation gives efficient resource use and makes updating or patching a service a clean, low-risk operation — replace the container image, the orchestrator rolls it in. This is also the foundation of the scalability story in Topic 2.

The stance

"Open" is concrete here: external systems use the same interfaces as IFS Cloud's own clients. There is no private API the product keeps for itself.

Two shapes

Point-to-point is direct and quick but the link count explodes — n systems, up to n-squared links. The digital switchboard centralises routing and transformation in one hub. Neither is universally right; the message is to choose deliberately and hold an integration strategy, rather than letting point-to-point links accumulate as debt.

Integrity

Critical point for data-governance reviewers: inbound integration data always passes through business logic. It is never inserted directly into tables. Bad data is rejected at the door, not discovered downstream.

Horizontal scale

Two levers. Add node machines to the cluster for more total capacity. Increase the replica count of a container to spread load across them. Both are routine operations — no re-architecture.

High availability by design

This is the key point. When Kubernetes runs more than two replicas of a container, it places them on different nodes. If a node fails, the surviving replicas keep serving and the orchestrator reschedules the lost ones. HA is a property of how the cluster is configured — not a separate product.

The principle

Horizontal scaling means capacity grows by adding identical units — nodes and replicas — not by forklift-upgrading to bigger servers. It scales in both directions, so the service is not over-provisioned.

Sizing

Environments are sized against five dimensions: solution complexity, user counts and concurrency, data handling and residency, availability tier, and custom/storage needs. This is a deliberate, validated design exercise — not guesswork.

Concurrency limit — explain it carefully

It is a performance guarantee, not a hard cap on named users. It is the number of users who can be simultaneously active, measured over an hour. It stabilises resource allocation and keeps the service rightsized. If the prospect has seasonal or shift-pattern peaks, this is where to discuss it.

Topic 02 · Resilience

High availability and resilience

Resilience is layered — the orchestrator, the Azure platform, and an optional high-availability tier each contribute.

LAYER 1
Orchestrator resilience
Kubernetes places replicas across nodes. A failed container is rescheduled automatically; a failed node sheds load to its peers. Self-healing is built in.
LAYER 2
Azure platform resilience
Compute, storage, network, and power are redundant within the data centre, with automatic recovery from low-level hardware faults — at both primary and secondary sites.
LAYER 3
High Availability service
An optional add-on for mission-critical operations — deployed across Azure Availability Zones in supported regions, including Australia East (Sydney).
Availability Zones
An Availability Zone is a physically separate facility within a region, with independent power, cooling, and network. The HA service spreads the deployment across zones so a single-facility failure does not take the service down. Australia East (Sydney) supports Availability Zones.
Outcome-based SLA
The managed cloud service commits to an application-level availability SLA, financially backed by service credits. The guarantee covers the whole stack working together — not individual components in isolation.

The model

Role-based access control built on Permission Sets. The non-negotiable rule: privileges reach a user only through a Permission Set, never by direct grant. Draw the chain — user, permission set, privileges.

Why that rule matters

It makes access auditable. A reviewer inspects a small number of well-named sets, not thousands of individual grants. It is the difference between an access review that is feasible and one that is not.

Data visibility

Function access is one axis; data visibility is the other. Company and Site constructs partition who sees which data — useful for a multi-entity telco and directly relevant to GDPR data-transfer restrictions.

Granularity

Access can be restricted to a low level of granularity where sensitive data demands it. History logging records every action for forensics. Segregation of duties gets its own slide — flag it and move on.

Default, not optional

The framing: encryption is the default state of the data. There is no unencrypted mode to opt out of.

In transit

TLS over HTTPS, RSA 2048-bit keys, Perfect Forward Secrecy so a future key compromise cannot decrypt past sessions. Traffic between Azure VMs is encrypted with SMB 3.0.

At rest

Keys

The detail security architects look for: keys live in Azure Key Vault, separate from the data, with restricted access. Encryption is only as strong as key custody — and key custody is handled correctly here.
